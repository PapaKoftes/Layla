A recurring root cause dominates this round: **bulk file tools validate only the operation root against `inside_sandbox()`, never each resolved file**, so `rglob` + `is_file()`/`read_text()`/`write_text()` follow in-sandbox symlinks to arbitrary out-of-sandbox targets (findings #5, #6, #7, #8). ### #1 — HIGH — `workspace_root` never confined to `sandbox_root` on the main `/agent/run` path; executor promotes it to the effective sandbox for every generic tool
- **File:** `agent/routers/agent.py:1434`
- **Root cause:** The request field `workspace_root` (schemas/requests.py:19, only max_length validated) flows unchecked through run_setup.py:444 (`workspace = workspace_root or sandbox_root`) into the run. For generic-dispatched tools, tool_dispatch.py:862 calls `core.executor.run_tool(..., sandbox_root=workspace)`, and executor.py:88-91 does `set_effective_sandbox(workspace)` on the tool thread. `sandbox_core._get_sandbox()` (sandbox_core.py:58-64) returns the thread-local effective sandbox in preference to config `sandbox_root`, so `inside_sandbox()` is evaluated against the caller-supplied workspace. Any generic write tool gating only via `inside_sandbox()` (create_svg general.py:966, create_mermaid general.py:977, notebook_edit_cell general.py:337, write_csv file_ops.py:1171, create_archive/extract_archive/merge_pdf/scan_repo/geometry writers) therefore writes anywhere under `workspace`. The intended invariant workspace_root ⊆ sandbox_root is enforced in understand_mode (agent.py:576) but omitted on the main run path. The dedicated write_file/apply_patch/replace_in_file handlers do NOT set the effective sandbox, so the same "write a file" operation has two inconsistent containment boundaries.
- **Fix sketch:** In the main /agent/run path (and in _dispatch_autonomous_run / run_setup where `workspace` is resolved at run_setup.py:444), validate `Path(workspace_root).resolve()` is inside config sandbox_root with the same inside_sandbox gate used at agent.py:576, and reject otherwise. Alternatively, have executor.run_tool intersect/validate the passed sandbox_root against config sandbox_root before set_effective_sandbox.
- **Failing input:** POST /agent/run (main autonomous path) with allow_write=true and workspace_root="C:\\" (or any dir outside sandbox_root). run_setup.py:444 sets workspace="C:\\"; agent invokes a generic write tool (create_svg/create_mermaid/write_csv/notebook_edit_cell/create_archive); tool_dispatch.py:861 calls run_tool(sandbox_root="C:\\"), executor.py:88-91 set_effective_sandbox("C:\\"), _get_sandbox() returns that root ahead of config, inside_sandbox("C:\\Windows\\Temp\\evil.svg") passes, general.py:971 writes outside sandbox_root. No inside_sandbox confinement exists on the main run path, unlike understand_mode.

### #2 — LOW — Shell command blocklist only inspects argv[0]; `bash -c` / `sh -c` / `python -c` trivially bypass the hard-block
- **File:** `agent/services/sandbox/shell_runner.py:56`
- **Root cause:** `_cmd_blocked` (shell_runner.py:56-70) derives the blocked check purely from the basename of argv[0] (same for system.py:57-60 and the `_SHELL_BLOCKLIST` contract in sandbox_core.py:92-97, documented "never allowed even with allow_run=True"). Because subprocess is invoked with an argv list, a blocked binary need only run as an argument to a permitted interpreter whose basename is not in the blocklist. `sh`, `bash`, `python`, `node`, `perl`, `env`, `xargs`, `find` are all absent from both the blocklist and (when shell_restrict_to_allowlist is off — the default) any allowlist, so `['bash','-c','rm -rf ~']` executes the very operations the blocklist is meant to hard-stop.
- **Fix sketch:** Treat interpreter/wrapper invocations specially: when argv[0] basename is a known shell/interpreter (sh/bash/zsh/python/perl/node/env/xargs/find/...), scan the -c/-e/-Command payload and remaining tokens for blocklisted basenames, or refuse `-c`/inline-code forms unless explicitly allowlisted. Prefer allowlist-by-default for the shell tool given it is already approval-gated.
- **Failing input:** shell(argv=['bash','-c','rm -rf $HOME/important'], cwd=<sandbox>) — or portably shell(argv=['python','-c','import shutil,os; shutil.rmtree(os.path.expanduser("~/important"))']). base='bash'/'python' is not in _DEFAULT_BLOCKLIST or _SHELL_NETWORK_DENYLIST (python is even in the restrict=True allowlist), _cmd_blocked returns None, injection patterns don't match, inside_sandbox only checks cwd, and run_shell_argv executes the rm.

### #5 — HIGH — `search_replace` writes through in-sandbox symlinks to files outside the sandbox
- **File:** `agent/layla/tools/impl/file_ops.py:509`
- **Root cause:** search_replace() validates only root_path against inside_sandbox() (line 501), then iterates root_path.rglob(file_glob) and, for every f where f.is_file() is true, calls f.write_text(new_content) at line 535 when dry_run is False. rglob yields symlink files, f.is_file() follows the link, write_text writes THROUGH the symlink. Each matched file f is never re-checked with inside_sandbox()/resolve(). Unlike write_file() (resolves each target and re-checks per file), the bulk path only checks the root.
- **Fix sketch:** Inside the rglob loop, after `if not f.is_file(): continue`, add `if not inside_sandbox(f): continue` (inside_sandbox resolves symlinks). Equivalently operate on f.resolve() and re-validate before any read_text/write_text. Apply the same gate to the read side (content = f.read_text) since it also leaks outside content.
- **Failing input:** Linux deployment: symlink `evil.txt -> /etc/crontab` exists inside the sandbox (pre-existing in a checked-out repo / node_modules, or planted). Agent calls search_replace(root='.', find='<string present in target>', replace='PWNED', file_glob='evil.txt', dry_run=False). root_path='.' passes line 501; rglob yields the symlink; f.is_file() follows to True; f.write_text() at line 535 writes through it, overwriting /etc/crontab. No per-file guard exists.

### #7 — MEDIUM — `grep_code` Python fallback reads content of in-sandbox symlinks pointing outside the sandbox
- **File:** `agent/layla/tools/impl/code.py:61`
- **Root cause:** grep_code() validates only root against inside_sandbox() (line 38). Its Python fallback (when ripgrep is absent) loops `for f in root.rglob(file_glob)` (line 61) and returns `f.read_text(...)` line content (lines 65-67) for any f where f.is_file() is true. rglob yields symlink files and f.is_file()/read_text follow the link, so matched lines of a file OUTSIDE the sandbox are returned. ### #8 — LOW — Analysis tools (dead-code/complexity/import scanners) read symlinked out-of-sandbox files during rglob
- **File:** `agent/layla/tools/impl/code.py:351`
- **Root cause:** Multiple read-only scanners in code.py (secret/dead-code scanner ~351-355, complexity ~465-473, unused-import ~500-504/518-554, syntax check ~616-619) validate only the target root with inside_sandbox(), then rglob(...) and read_text() every matched file with no per-file re-check. rglob + is_file()/read_text follow symlinks, so a symlink inside the sandbox pointing outside is opened and folded into the returned analysis (line snippets, matched secret patterns, source excerpts), leaking out-of-sandbox content. - **Fix sketch:** Centralize a helper `_iter_sandbox_files(root, glob)` that yields only files where inside_sandbox(f) holds (symlink-resolved), and route every rglob-based reader/writer (grep_code, search_replace, rename_symbol, and these analysis scanners) through it, so the symlink-safe guarantee extends from single-file tools to bulk tools.
- **Failing input:** POSIX: inside the sandbox create symlink `leak.env -> /home/user/.aws/credentials`. Call security_scan(path=".", scan_type="secrets"). target="." passes inside_sandbox; scan_files via target.rglob("*") yields leak.env; f.is_file() follows to True; suffix ".env" allowed; ".git" filter passes; fpath.read_text() reads the credentials file; matched secret patterns (AWS Access Key, etc.) returned in `findings` with 80-char snippets. Same escape applies to code_metrics/dependency_graph/find_todos/code_symbols/code_lint.

### #9 — HIGH — Remote `/settings` write can flip `plugins_enabled` (the plugin-code-execution consent gate) because it is missing from `_REMOTE_PROTECTED_KEYS`
- **File:** `agent/routers/settings.py:29`
- **Root cause:** The plugin-RCE defense rests on plugins_enabled (default False) being an explicit operator action: plugin_loader.py:31/101-112 only exec_module()s a plugin's tools.py when plugins_enabled is true, and mcp_client.py:74 only spawns plugin-declared MCP subprocess servers when true. The settings router defines _REMOTE_PROTECTED_KEYS (settings.py:29-49) to stop remote clients changing security-critical keys — and the comment (settings.py:41-42) explicitly says a remote write "must not point plugin/MCP/model loading at attacker-staged files" — protecting plugins_dir, mcp_stdio_servers, mcp_client_enabled, safe_mode, sandbox_root. But plugins_enabled itself — the master switch — is NOT in the set. It is in EDITABLE_SCHEMA (config_schema.py:130) and sync_save_settings persists editable keys (route_helpers.py:37), so POST /settings {"plugins_enabled": true} passes the remote filter (settings.py:477-480) and is written to runtime_config.json. /settings is on the remote interactive allowlist (main.py:1160), reachable by any bearer-authenticated non-loopback client (main.py:1198-1254). Asymmetry: safe_mode (also category 'safety', also a code-exec gate) IS protected while its sibling plugins_enabled is not — a clear omission.
- **Fix sketch:** Add 'plugins_enabled' (and consider 'agent_hooks_enabled', 'hooks_require_allow_run', 'skill_venv_enabled') to _REMOTE_PROTECTED_KEYS in settings.py so a non-local caller cannot enable plugin/hook code execution.
- **Failing input:** With remote_enabled=true and remote_mode=interactive, a bearer-authenticated non-loopback client sends POST /settings {"plugins_enabled": true}. is_direct_local() is false; blocked = _REMOTE_PROTECTED_KEYS.intersection(body) is empty (plugins_enabled not in the frozenset settings.py:29-49), so the key is not popped and reaches save_config_keys(editable_only=True) — plugins_enabled is in EDITABLE_SCHEMA — and is persisted. On the next plugin (re)load (60s TTL, restart, or system_doctor) plugin_loader.py:31/101-117 exec_module()s any tools.py in the default plugins/ dir and mcp_client.py:74 spawns plugin-declared MCP servers. Precondition: a plugin with tools.py/MCP must already exist in the (still-protected) plugins_dir; the remote can only flip the consent gate, not stage new code.

### #10 — HIGH — `fetch_url` tool (default fetch path) uses raw urlopen with no redirect/rebinding guard
- **File:** `agent/layla/tools/web.py:152`
- **Root cause:** fetch_url() pre-checks the URL once with _is_safe_url() (delegates to hardened url_guard, resolving DNS) then fetches via raw urllib.request.urlopen(req) using urllib's DEFAULT redirect handler. The pre-check is TOCTOU: (a) urlopen re-resolves the hostname, so a domain resolving to an internal IP on the second lookup (DNS rebinding) is fetched; (b) urlopen silently follows 3xx with no per-hop validation, so a public URL that 302s to http://169.254.169.254/ or http://127.0.0.1/ is fetched. The codebase already has services.safety.url_guard.safe_urlopen (re-validates initial URL AND every redirect hop); sibling tools http_request (impl/web.py:273) and check_url (impl/web.py:445) were migrated to it, but this fetch_url — the primary, default-reachable fetch tool — was not. It is the default path: fetch_url_tool (impl/web.py:66) delegates here whenever crawler_backend is auto/basic (the default).
- **Fix sketch:** Replace the raw urllib.request.urlopen call in fetch_url with services.safety.url_guard.safe_urlopen(req, timeout=_TIMEOUT) (keep existing headers). This re-validates every redirect hop and closes the rebinding window, matching the already-migrated http_request/check_url.
- **Failing input:** fetch_url("http://attacker.test/r") where attacker.test/r responds 302 Location: http://169.254.169.254/latest/meta-data/ (or http://127.0.0.1:PORT/ / http://192.168.1.1/). _is_safe_url passes on the public host; urllib's default HTTPRedirectHandler follows the 302 to the internal target with no per-hop check and returns its body, surfaced to the LLM and KB-ingestible. Registered with require_approval:False and is the default crawler_backend=auto/basic path. Secondary: DNS rebinding TOCTOU on a low-TTL attacker domain.

### #11 — LOW — `web_crawler.crawl_url` performs no SSRF validation on any backend (basic backend = raw urlopen)

### #14 — LOW — `/v1` streaming path emits model tokens live before `check_output` runs, so Tier-1 output is not filtered for non-browser clients
- **File:** `agent/routers/openai_compat.py:462`
- **Root cause:** In the /v1 streaming branch, model tokens are yielded incrementally (openai_compat.py:358-366) with only stream_safe_prefix marker-cleaning and stop-sequence handling; content_guard check_output runs only afterward on the accumulated persisted copy (lines 462-474, comment acknowledges "the streaming tokens already emitted live"). The done-frame rewrites the STORED text but cannot retract already-streamed SSE chunks, and unlike /research (which sets a `blocked` flag at research.py:531-532 for research.js to replace the tokens) the OpenAI SSE contract has no equivalent, so an OpenAI-SDK / raw HTTP consumer receives the unsafe content verbatim.
- **Fix sketch:** Run check_output incrementally on the accumulated stream_reason buffer before yielding each delta (block/replace once a Tier-1/2 match appears), or buffer output and guard before the first emit; at minimum emit a terminal replacement frame when the persisted-copy check trips so streaming clients can discard prior chunks.
- **Failing input:** POST /v1/chat/completions with stream=true and a jailbreak prompt that evades the input content-guard (lines 236-262) yet induces the local model to emit Tier-1 content token-by-token. The tokens are streamed verbatim via lines 358-366; check_output only runs afterward at lines 469-472 on the persisted copy, and the empty-delta done-frame cannot retract already-sent SSE chunks.
