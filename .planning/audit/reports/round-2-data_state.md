## Report-only findings

### 1. [HIGH] Corrupt/partially-written knowledge_graph.graphml is silently overwritten with an empty graph — total, unrecoverable knowledge-graph loss
- **File:** `agent/layla/memory/memory_graph.py:32`
- **Root cause:** `_get_graph()` (lines 25-33) wraps `nx.read_graphml(GRAPH_PATH)` in a bare `except Exception: pass` and, on ANY read failure, falls through returning a fresh empty DiGraph. The callers `add_node` (line 142), `add_edge` (line 151), and `save_graph` (line 88) then unconditionally call `_save_graph(G)`, which atomically `os.replace()`s the existing on-disk file with the near-empty in-memory graph. So a single unreadable read — from a power-loss partial write, transient IO error, encoding/parse issue, or a leftover truncated file — causes the next write to permanently destroy the entire knowledge graph. Unlike `layla.db` (which has `verify_and_recover_db` to quarantine a corrupt file and restore the newest backup), the graphml has NO integrity gate, NO quarantine (`.corrupt` move-aside), and is NOT included in `db_backup.backup_database` (which only backs up `layla.db` + the Chroma/vector store — confirmed by grep: `knowledge_graph.graphml` appears in no backup path), so there is no backup to restore from either.
- **Fix sketch:** In `_get_graph`, distinguish 'file missing' (fresh install → empty is correct) from 'file exists but read_graphml raised' (corruption → do NOT return empty and do NOT let callers overwrite). On a parse failure, move the bad file aside (`GRAPH_PATH + '.corrupt'`) and raise/refuse to overwrite, mirroring `verify_and_recover_db`. Also add `knowledge_graph.graphml` to `db_backup` so a backup exists to restore.
- **Failing input:** Precondition: `knowledge_graph.graphml` exists but `nx.read_graphml()` raises — from a leftover truncated/zero-length file after a power-loss (`os.replace` metadata flushed but data blocks not fsync'd), OR any transient read error (Windows sharing violation from a concurrent reader, a permission/IO blip, or an XML parse/encoding error). On the next memory write (e.g. `add_node('foo')` from the scheduler memory job or drone consolidation worker): `_get_graph()` catches the exception at `memory_graph.py:32-33`, returns an empty DiGraph; `existing=[]` so `node_id=max([],default=-1)+1=0`; `_save_graph(G)` at line 142 atomically `os.replace()`s `GRAPH_PATH` with a 1-node graph. Every prior node and edge is permanently gone.

### 3. [LOW] graphml/config/codex atomic writers os.replace without fsync — power loss can leave a zero-length file despite the 'atomic' rename
- **File:** `agent/layla/memory/memory_graph.py:57`
- **Root cause:** `_save_graph` writes the temp file with `nx.write_graphml` then `os.replace()`s it, but never `fsync()`s the temp file's data (nor the containing directory) before/after the rename. `os.replace` guarantees rename atomicity, NOT data durability: on power loss the metadata rename can be committed while the file's data blocks are still unflushed, yielding a zero-length or garbage graphml after reboot. The repo already has a durable helper (`_fsync_file` in `install/model_downloader.py`, used before its `os.replace` at line 464) but the memory/graph/codex/config writers (`memory_graph.py:57-58`, `relationship_codex.py:58-60`, `runtime_safety.py:165-173`) don't fsync. On its own this is low, but it is the concrete mechanism that produces the corrupt graphml which finding #1 then catastrophically overwrites — the two chain into total knowledge-graph loss on a single power event.
- **Fix sketch:** In `_save_graph`, open the temp file, write, `os.fsync(fd)` (and ideally fsync the directory) before `os.replace`. Reuse the existing `_fsync_file` helper pattern. Apply the same to the codex/config atomic writers.
- **Failing input:** Power loss (or kernel panic / hard reset) occurs after `os.replace` records the tmp->knowledge_graph.graphml rename in the directory entry but before the temp file's data pages are flushed to disk (e.g. ext4 default data=ordered with delayed allocation). On reboot `knowledge_graph.graphml` exists but is 0-length or truncated; `nx.read_graphml` in `_get_graph` then raises, is swallowed by the bare except (line 32-33), and the graph silently reverts to empty.

### 4. [HIGH] Crash-reaped missions become permanently unresumable ('interrupted' is a dead-end status)
- **File:** `agent/layla/memory/missions_db.py:280`
- **Root cause:** `reap_orphaned_tasks()` resets crashed 'running' missions to `status='interrupted'` (`missions_db.py:280-281`), promising in its docstring that 'the user resumes manually'. But NO code path accepts 'interrupted' as a resumable state: `resume_mission_api` rejects anything not in `('paused','pending')` (`routers/missions.py:97-98`), `run_mission()`/`resume_mission()` require `('pending','paused')` (`mission_manager.py:75`), and `get_active_missions()` only fetches 'running'/'pending' (`missions_db.py:158-159`) so the mission_worker never picks it up either. The mission_worker docstring even claims it 'Persists progress for restart recovery' (`scheduler/jobs.py:111`) — the reaper directly defeats that recovery. A mission that was mid-flight (e.g. step 3 of 10, fully persisted with plan/current_step/results) during any crash or restart is stuck forever with no resume path.
- **Fix sketch:** Either mark reaped missions 'paused' (which IS resumable and re-executes from current_step, not from scratch — missions are step-idempotent via current_step so the side-effect-safety rationale for 'interrupted' is weaker for missions than for background_tasks), or add 'interrupted' to the accepted set in `resume_mission_api`/`run_mission` and to `get_active_missions`. Also fix the board bucketing (see finding #5).
- **Failing input:** Create a mission via POST /mission (goal with a multi-step plan). The scheduled mission_worker (`registry.py:94`, every ~2 min) advances it; `run_mission` sets status='running' and current_step/results persist as it goes (e.g. step 3/10). Kill/restart the server. On startup `main.py:222` calls `reap_orphaned_tasks()`, which (`missions_db.py:280-281`) sets the mission's status to 'interrupted'. Now: POST /mission/{id}/resume returns 400 'cannot resume mission in status interrupted'; `run_mission`/`resume_mission` return False; and `get_active_missions` only selects status IN ('running','pending'), so the worker never repicks it. The only lifecycle move left is /cancel (sets status='failed'). The mission is permanently unresumable mid-flight.

### 5. [MEDIUM] 'interrupted' missions are silently mislabeled as 'backlog' on the Kanban board
- **File:** `agent/routers/missions.py:149`
- **Root cause:** `mission_board_api()` has no case for `status='interrupted'`; it falls through the else branch (`routers/missions.py:149-150`) and is bucketed into 'backlog' alongside genuine 'pending' missions. A user seeing a crash-interrupted mission in 'backlog' will click resume (the natural action for backlog/pending items), which then fails with a 400 because resume rejects 'interrupted'. The reaper's outcome is thus both invisible and misleading — the mission looks like fresh backlog work but cannot actually be started.
- **Fix sketch:** Add an explicit bucket/label for 'interrupted' (e.g. a 'needs_attention' column) or, once resume accepts it, surface it distinctly from fresh 'pending' backlog.
- **Failing input:** Server crashes while a mission is 'running'. On next startup `main.py:222` calls `reap_orphaned_tasks()`, setting the row to status='interrupted' (`missions_db.py:280`). GET /missions/board then returns it under board.backlog / counts.backlog via the else branch (`missions.py:149-150`), indistinguishable from a genuine 'pending' mission. A subsequent POST /mission/{id}/resume returns HTTP 400 because resume accepts only ('paused','pending') (`missions.py:97-98`).

### 6. [LOW] 'queued' background tasks are never reaped after a crash (reaper only covers 'running')
- **File:** `agent/layla/memory/missions_db.py:281`
- **Root cause:** `reap_orphaned_tasks()` only resets rows WHERE status IN ('running','started','in_progress') (`missions_db.py:281`). A background task is durably written with status='queued' at enqueue time (`agent_task_runner.py:359` `"status": "queued"`, persisted via `save_background_task` at :371) BEFORE its daemon thread flips it to 'running' (`_run_background_task` sets 'running' at :642-643). If the process dies after the DB row is saved but before/at the moment the thread starts (or if the thread never got scheduled), the row stays 'queued' forever. Background tasks are in-memory daemon threads (`_TASKS` is process-local, `agent_task_runner.py:18`) that are lost on crash — nothing re-runs or fails a 'queued' row on restart, and the reaper skips it. `list_background_tasks`/UI then shows it 'queued' indefinitely.
- **Fix sketch:** Add 'queued' to the reaper's WHERE clause for the background_tasks table (a queued task never started, so marking it 'interrupted'/'failed' is safe — no side effects ran), or explicitly fail queued rows on startup since their in-memory thread cannot be recovered.
- **Failing input:** Enqueue any threaded or subprocess autonomous background task (POST to the agent-task endpoint). `save_background_task()` at `agent_task_runner.py:371` commits the row with status='queued' (`missions_db.py:207`) BEFORE `th.start()` (:380). Kill the process in the window before the worker's `update_background_task(status='running')` commits (:647 threaded / :451 subprocess) — or if that DB write silently fails (it is try/except-logged). On restart, `reap_orphaned_tasks()` (wired at `main.py:221-222`) resets only status IN ('running','started','in_progress') (`missions_db.py:281`), skipping the row, which stays 'queued' forever with no other resume path.

### 8. [LOW] memory_conflicts table is append-only with no retention, cap, or resolved-row cleanup
- **File:** `agent/services/memory/consistency_guard.py:96`
- **Root cause:** `record_conflict()` does an unconditional `INSERT INTO memory_conflicts (... created_at, resolved) VALUES (?,?,?,?,?,?,0)` for every detected contradiction, called from `check_and_flag()` on each saved learning (gated by `memory_consistency_guard_enabled`, default True). The table has a `created_at` column but is NOT listed in `apply_retention_policies`' `policies` or `hard_caps` (`memory_consolidation.py:95-133`), and nothing deletes rows — not even ones marked resolved. It therefore grows monotonically with the number of contradictions ever detected.
- **Fix sketch:** Add `memory_conflicts` to `apply_retention_policies` (age-based on created_at, e.g. `retention_memory_conflicts_days`), and/or delete rows where resolved=1 older than a short window.
- **Failing input:** Long-running instance where saved learnings repeatedly trip the contradiction heuristic (e.g. numeric value-drift like "timeout is 30s" then "timeout is 60s", or negation flips on near-identical statements): each detection appends a memory_conflicts row that is never deleted, and resolving a conflict only sets resolved=1 without reclaiming the row, so the table grows without bound.

### 9. [LOW] Canonical SQLite knowledge graph (entities/relationships) has no decay or hard cap, unlike the NetworkX graph
- **File:** `agent/layla/memory/data_migrations.py:471`
- **Root cause:** The authoritative `entities` and `relationships` tables (`data_migrations.py:471` and :490), written via `memory_router.upsert_entity`/`upsert_relationship`, have no retention policy, confidence decay, or hard cap anywhere. Content-addressed SHA256 ids (`schemas/entity.py:147/212`) dedup exact repeats so the same entity/relationship upserts in place, but the count of DISTINCT entities/relationships discovered from conversations, code indexing, and codex scans grows monotonically forever. This is inconsistent with the NetworkX knowledge graph, which IS bounded (`memory_graph.py:91` `_prune_graph_if_needed` caps at `knowledge_graph_max_nodes=5000` and evicts oldest), and with learnings, which decay (`memory_consolidation.py:261`). The 'single source of truth' graph is the one store with no bound at all.
- **Fix sketch:** Add last_seen_at-based decay/eviction or a hard row cap for entities/relationships (mirroring the 5000-node NetworkX cap), pruning low-confidence, long-unseen entities and cascading relationships via the existing ON DELETE CASCADE.
- **Failing input:** (growth-over-time; no single-shot repro) monotonic accumulation of distinct entities/relationships across conversations and code/codex scans with no eviction path.

### 11. [LOW] repo_indexer.migrate() has no schema versioning and only CREATE TABLE IF NOT EXISTS, so future column changes silently never apply to existing repo_index.db
- **File:** `agent/services/workspace/repo_indexer.py:53`
- **Root cause:** `repo_indexer.py` maintains its OWN independent persistent SQLite DB (repo_files/repo_symbols/repo_imports/repo_calls) with a process-level `_MIGRATED` guard (line 32/56/104) but NO schema_version table, NO ALTER-TABLE backward-compat path, and only `CREATE TABLE IF NOT EXISTS` DDL (lines 59-101). Because `CREATE TABLE IF NOT EXISTS` is a no-op against an already-existing table, any later addition/rename of a column to these definitions will silently fail to apply to any pre-existing repo_index.db on an operator's machine, producing 'no such column' errors at query time. This is the exact schema-drift pattern the main memory DB explicitly guards against (`migrations.py` adds columns via idempotent ALTER at 291-383 and stamps schema_version at 957-967, plus a downgrade guard at 90-104); repo_indexer has none of that.
- **Fix sketch:** Give repo_indexer the same minimal drift protection as the memory DB: a schema_version row and additive `ALTER TABLE ... ADD COLUMN` steps guarded by duplicate-column handling, run whenever schema evolves. Low severity today because the code symbol index is rebuildable and the schema is currently stable, but it is a genuine backward-compat gap for the next schema change.
- **Failing input:** None at HEAD — the defect is latent. Reproducing it requires a hypothetical future schema change (e.g. adding `repo_symbols.complexity`). No current query/insert references a column absent from the CREATE TABLE DDL, so no input reproduces a 'no such column' error against today's code.

### 12. [MEDIUM] Auto-compact background thread silently drops conversation turns appended during the LLM summarization window
- **File:** `agent/shared_state.py:124`
- **Root cause:** `append_conv_history()` spawns a daemon thread (`_compact_bg`) on every assistant message. That thread snapshots the deque under `_conv_hist_lock`, RELEASES the lock, runs a slow LLM summarization (`maybe_auto_compact` -> `summarize_history` -> `_compress_to_summary` calls `run_completion`, seconds on the CPU-only box), then re-acquires the lock and does `h1.clear()` followed by repopulating the deque from the STALE snapshot's compacted result. Any user/assistant messages appended to the same conversation deque during the summarization window are wiped by `clear()` and are NOT present in the snapshot, so they are permanently lost from the in-memory history that feeds prompt-building on subsequent turns. There is no compare-and-swap / re-merge of messages appended after the snapshot, and every assistant append spawns another compaction thread so overlapping compactions also clobber each other (last writer wins). The `clear()`+repopulate happens even when compaction is a no-op (after >= before), so it can drop concurrent messages without shrinking anything.
- **Fix sketch:** Do not `clear()`+repopulate from a pre-LLM snapshot. Either (a) recompute the tail to preserve: capture the snapshot length, and after summarization only replace the exact prefix that was summarized while re-appending any messages the deque gained since the snapshot; or (b) guard with a monotonically-increasing per-conversation version and skip the swap if the deque changed during the window; or (c) run compaction on a copy and merge, never wiping newer entries.
- **Failing input:** Two turns sharing conversation_id C (e.g. a background `agent_task_runner` turn + an interactive turn, or two fast successive turns). Turn 1's assistant append spawns `_compact_bg` thread A, which snapshots the deque as `[.., u1, a1]` (`shared_state.py:131-133`) and RELEASES `_conv_hist_lock`. A then enters `maybe_auto_compact` -> `summarize_history` -> `_compress_to_summary`, which (LLM idle, threshold exceeded) grabs busy_lock and runs `run_completion` for several seconds. During that window turn 2's request appends 'user' u2 via `append_conv_history` (needs only the brief `_conv_hist_lock`, NOT busy_lock), so the deque becomes `[.., u1, a1, u2]`. A's LLM completes; its 'compacted' result derives solely from the stale snapshot and never contained u2. A re-acquires `_conv_hist_lock`, calls `h1.clear()` (line 139) and repopulates from compacted (140-141) -> u2 is gone. Turn 2 then generates and appends a2, leaving an orphan assistant turn a2 whose prompting user message u2 is permanently absent. If turn 2 completes both appends before A re-acquires the lock, both u2 and a2 are dropped.

### 13. [LOW] Local llama_cpp non-stream timeout is ineffective and stalls all LLM callers: ThreadPoolExecutor `with` block joins the hung worker while holding llm_serialize_lock
- **File:** `agent/services/llm/inference_router.py:327`
- **Root cause:** `run_completion_llama_cpp` implements a completion timeout by submitting `create_completion` to a ThreadPoolExecutor and calling `fut.result(timeout=...)`. But the executor is used as a context manager (`with _cf.ThreadPoolExecutor(max_workers=1) as ex:`), and the whole block is nested inside `with _llm_lock:`. On TimeoutError the code `return`s from INSIDE both `with` blocks, which triggers `Executor.__exit__` -> `shutdown(wait=True)`. `shutdown(wait=True)` BLOCKS until the still-running `create_completion` worker finishes. So the timeout never actually abandons a hung/slow generation — the request thread blocks for the full generation duration anyway, and it does so while STILL holding `llm_serialize_lock` (the single global LLM serialization point). Every other chat/agent/background LLM request that needs that lock is blocked for the entire hang. `invalidate_llm_cache()` is also called (line 344) while the worker is mid-eval on the same Llama instance.
- **Fix sketch:** Do not use the executor as a `with` block (which force-joins). Create the executor without a context manager and on timeout call `ex.shutdown(wait=False, cancel_futures=True)` and return immediately so the lock is released, letting the daemon worker die on its own; or drop the executor entirely and enforce timeouts via the backend's own generation limits. Also avoid `invalidate_llm_cache()` while a worker still holds a live reference to the instance.
- **Failing input:** A local GGUF non-streaming completion (`run_completion` with default stream=False) whose `create_completion` stalls or runs well past `llm_local_timeout_seconds`. `fut.result` raises `concurrent.futures.TimeoutError` at the budget, but the `return {...}` on line 348 executes inside `with ThreadPoolExecutor(...) as ex` (line 328) which is inside `with _llm_lock` (line 327). `Executor.__exit__` runs `shutdown(wait=True)`, joining the still-running uninterruptible C-level worker, so the function does not return until the full generation completes — and `llm_serialize_lock` (the global RLock) stays held the whole time.

### 14. [LOW] Streamed replies cannot be interrupted on client disconnect; the abandoned generation keeps llm_serialize_lock held, blocking all other LLM requests
- **File:** `agent/routers/agent.py:1094`
- **Root cause:** The streaming handlers create a `threading.Event` (`client_abort` / `_client_abort_fast`) and start `_watch_client_disconnect` to set it when the HTTP client disconnects, but `stream_reason()` has NO cancellation parameter (`services/agent/stream_handler.py:44-58` takes no client_abort_event) and the token-producer worker never checks the event. So on disconnect the event flips but the underlying LLM streaming loop keeps generating tokens to completion. Because `run_completion_llama_cpp` holds `llm_serialize_lock` for the entire streaming loop (`inference_router.py:294` `with _llm_lock:` around the whole `for chunk in ...` yield loop), an abandoned/disconnected stream continues to hold the single global LLM serialization lock until it finishes on its own, stalling every concurrent chat/agent/background LLM caller waiting on that lock.
- **Fix sketch:** Thread the `client_abort` `threading.Event` through `stream_reason` into `run_completion`/`run_completion_llama_cpp` and break out of the `for chunk in create_completion(...)` loop when the event is set (releasing the lock), and have the router's token-consumer loop stop pulling/generating once client_abort is set.
- **Failing input:** User A sends a self-contained question (fast path) or a stream_pending reply, then closes the tab mid-generation. `_watch_client_disconnect` sets `_client_abort_fast`/`client_abort`, but the daemon token-producer thread (`_worker`/`_stream_worker` → `stream_reason` → `run_completion(stream=True)`) never checks the event and keeps generating to max_tokens while holding the global `llm_serialize_lock` (`inference_router.py:294`). A concurrent background LLM task (or User B) blocks on that lock until A's abandoned generation finishes on its own (~tens of seconds on the CPU-bound box).
