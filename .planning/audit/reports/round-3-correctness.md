### #1 — HIGH — Three memory blocks silently dropped: `MEMORY_SECTION_ORDER` missing `working_memory` / `answer_feedback` / `golden_examples`
- **File:** `agent/services/context/context_merge_layers.py:8`
- **Root cause:** `build_system_head` writes 20 keys into the `memory_sections` dict, but `memory_block` is materialized ONLY by iterating the canonical tuple `MEMORY_SECTION_ORDER` (system_head_builder.py:1182 and the except-fallback :1186 both iterate exclusively over it). That tuple (context_merge_layers.py:8-26) is missing three keys that `build_system_head` sets: `working_memory` (system_head_builder.py:1008), `answer_feedback` (:1150), and `golden_examples` (:1171). Any key not in the tuple is never emitted — computed then silently discarded before assembly. The module docstring even states "Keys must match those used in ... build_system_head", so this is a coverage bug, not intent. `answer_feedback` is the BL-242 feature folding explicit user 👎 + written corrections back into the prompt "so the next turn honours them" — retrieved then thrown away, so the feedback loop is inert. `working_memory` (running scratchpad of facts) never reaches the model. `golden_examples` few-shot exemplars never reach the model.
- **Fix sketch:** Add `working_memory`, `answer_feedback`, `golden_examples` to `MEMORY_SECTION_ORDER` at their intended precedence positions (working_memory near aspect_memories/learnings; answer_feedback near rl_feedback; golden_examples near reasoning_strategies). Add a guard/test asserting `set(keys ever written) ⊆ MEMORY_SECTION_ORDER` to prevent regressions.
- **Failing input:** A substantive large-model turn (not `_skip_expensive`, not `_small_model` — i.e. `n_ctx > 4096` and not a small model) where the user previously gave a thumbs-down plus written correction. `feedback_hint_for_prompt()` returns text stored at system_head_builder.py:1150 under `memory_sections['answer_feedback']`, but that key is absent from `MEMORY_SECTION_ORDER`, so lines 1182/1186 never emit it. `working_memory` (set :1008, unconditional) and `golden_examples` (set :1171) are dropped identically on every qualifying turn.

### #2 — LOW — Memory precedence inverted: memory block relocated AFTER knowledge (Reference docs) when `knowledge_graph` is present
- **File:** `agent/services/context/context_manager.py:388`
- **Root cause:** In `build_system_prompt` the dedup rebuild (lines 388-396) treats `{'memory','knowledge_graph'}` specially: it splits `built` into `non_mem` + a single merged memory block appended at the END. But the declared canonical order (lines 284-301) places `memory` and `knowledge_graph` BEFORE `knowledge`. So when both `memory` and `knowledge_graph` survive the budget (graph_associations non-empty), the merged block moves to after the `knowledge` (Reference docs) section — inverting intended precedence where recalled memories/graph outrank static reference docs. Reorder only triggers when `len(mem_contents) > 1`, so it is intermittent (present iff a knowledge_graph block exists), producing non-deterministic section ordering across turns.
- **Fix sketch:** Rebuild preserving position: replace the first memory-key slot in `built` with the merged block and drop the second, instead of appending `merged_mem` to the tail. Or dedup in place and re-sort `built` by the canonical `order` index after merging.
- **Failing input:** A single `build_system_prompt` turn where `sections["memory"]`, `sections["knowledge_graph"]`, AND `sections["knowledge"]` are all non-empty and all survive the token budget. `mem_contents` then has length 2, so the dedup branch (390-396) fires; `non_mem` still contains `("knowledge", "Reference docs:...")` in its canonical slot, and `built` is rebuilt as `non_mem + [("memory", merged_mem)]` — merged memory/graph emitted AFTER "Reference docs". On an otherwise identical turn where `graph_associations` is empty (`len==1`), no reorder occurs and memory stays before knowledge — order flips between turns.

### #3 — MEDIUM — `think` and `none` decision actions loop with no per-run cap — only wall-clock timeout bounds them
- **File:** `agent/services/agent/decision_loop.py:244`
- **Root cause:** In the main decision loop, the `think` branch (`action == "think"`, 244-269) and the `none` branch (`intent == "none"`, 333-339) each append a step and `continue` WITHOUT incrementing any progress counter. `state["depth"] += 1` happens only on the tool-dispatch path (:499); the loop-stop counters `tool_calls`/`blocked_calls` (checked 111-113) are only touched by tool execution and guard rejections. The `while state["depth"] < 5` bound (:64) is therefore never advanced by think/none iterations. Loop-detection and duplicate guards (tool_guards.py, push_and_evaluate in tool_loop_detection.py which early-returns None for non-tool actions) are only reached on the tool path — think/none `continue` before `_run_tool_guards` (:435). `_think_seq` is incremented (:246) but never compared against any cap. Net: only `max_runtime` terminates a repeated think/none stream (default 900s agent / 90s light-chat, run_setup.py:458/503).
- **Fix sketch:** Add a per-run cap on non-executing decision actions: bound consecutive/total `think` via `_think_seq` and a `none` counter, forcing `intent="reason"` once a small threshold (3-4) is exceeded, mirroring `blocked_calls`'s `_blocked_cap` backstop. Or increment `depth` (or a dedicated `no_op_steps` counter) on the think/none continue so a bound actually terminates the spin.
- **Failing input:** An agentic run (`allow_write=True` or `allow_run=True`, so both self-contained-question fast paths at 212-218 and 296-304 are disabled) whose local decision model repeatedly emits `{"action":"think","thought":"..."}` (or `{"action":"none"}`). Each iteration hits 244-269 (or 333-339), appends a step and `continue`s without advancing `depth`/`tool_calls`/`blocked_calls`. The run spins, paying one decision-model completion per iteration, producing no answer, until `time.time() - start_time > max_runtime` (:99) trips at 900s (90s light-chat). `parse_decision` (decision_schema.py:134-139) preserves the actions, so this is model-reachable.

### #4 — HIGH — Possessive personal-data IMPERATIVES misclassified as self-contained → tools suppressed, answer hallucinated
- **File:** `agent/services/agent/response_builder.py:166`
- **Root cause:** `is_self_contained_question()` defaults to `return True` (:168), and its only guard against personal-data requests is `re.search(r"\b(my|our)\b", gl) and "?" in gl` (:157, :166) — it requires a QUESTION MARK. A possessive IMPERATIVE referencing personal data that doesn't match a hard-coded phrase list falls straight through to `return True`. The phrase lists only cover a few nouns (`my note`, `my todo`, `my goal`, `my plan`); emails, calendar, meeting notes, files, messages are absent. So `summarize my emails`, `check my calendar`, `list my meeting notes` are classified self-contained — violating the contract the unit tests assert for the interrogative form (`What's in my todo list?` must be False, tests/test_self_contained_question.py:30).
- **Fix sketch:** Drop the `and "?" in gl` qualifier so a possessive personal-data referent gates regardless of interrogative form, OR replace the default `return True` with a conservative default plus an explicit allowlist of genuinely self-contained shapes. Add emails/calendar/messages/inbox/files/notes to the personal-data signal set.
- **Failing input:** `summarize my meeting notes`

### #9 — MEDIUM — Default auto-tune silently disables the governor's foreground CPU-thread throttle
- **File:** `agent/services/llm/llm_gateway.py:482`
- **Root cause:** The model-load path only lets the ResourceGovernor throttle inference threads when `n_threads` is NOT already set in config: `if cfg.get("resource_governor_enabled", True) and not cfg.get("n_threads"):` (:482). The guard's stated intent (comment 477-481) is that an operator-pinned `n_threads` should win, so the governor only throttles when the operator left it auto. But auto_tune is on by default and AUTHORITATIVELY writes `n_threads` into the effective config: `PROFILE_KEYS` includes `"n_threads"`/`"n_threads_batch"` (auto_tune.py:49), `compute_optimization_profile` seeds from `get_recommended_settings` which always returns `n_threads = max(1, cpu_physical)` (hardware_detect.py:342, always ≥1/truthy), and `apply_auto_tune` copies it into config (auto_tune.py:261-264). `load_config()` applies this overlay (runtime_safety.py:868-871) and llm_gateway reads that same cfg (:406). Result: `cfg.get("n_threads")` is always truthy on a default install, so `not cfg.get("n_threads")` is dead and `get_inference_threads()` (resource_governor.py:358) never runs. The governor cannot tell an auto_tune-injected value from an operator-pinned one.
- **Fix sketch:** Let the governor own `n_threads` under auto-tune: either drop `"n_threads"`/`"n_threads_batch"` from auto_tune `PROFILE_KEYS` so the load path's governor branch is reachable, or change the llm_gateway gate to distinguish an operator pin from an auto-tuned value (e.g. only treat `n_threads` as pinned when it appears in `auto_tune_locked_keys` or in the raw on-disk `runtime_config.json`, not in the auto_tune-overlaid effective cfg).
- **Failing input:** Fresh default config on a CPU-only (potato/cpu tier) laptop: `auto_tune_enabled` defaults True, `resource_governor_enabled` defaults True, `n_threads` unset, `auto_tune_locked_keys` unset. User is actively typing (governor forced to WHISPER via `mark_user_active`). First model load occurs. Expected: `get_inference_threads()` returns `cores//2` to leave CPU headroom for the foreground user. Actual: `apply_auto_tune` has already injected `n_threads=max(1,cpu_physical)` into cfg, so llm_gateway.py:482's `not cfg.get("n_threads")` is False, the throttle is skipped, and Llama is constructed with every physical core.
